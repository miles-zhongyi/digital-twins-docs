# 4G srsTwin — Structure and Implementation

This chapter expands the 4G LTE twin documentation with system structure,
message flow, startup procedures, and known deviations from a production
network. It complements [`lte_digital_twin.md`](lte_digital_twin.md) and the
project [Glossary](../glossary.md).

## Startup

From `srsTwin/integration`:

```bash
docker compose -f docker-compose.4g.yml up -d --force-recreate
```

Wait ~30s, then verify attach:

```powershell
docker logs srstwin_ue4g 2>&1 | Select-String "Network attach successful"
docker logs srstwin_epc 2>&1 | Select-String "Attach request|UE Authentication Accepted|Attach Complete"
```

Dashboard (optional):

```bash
cd integration/dashboard
docker cp srstwin_ue4g:/tmp/ue4g.log logs/ue4g.log
docker cp srstwin_enb:/tmp/enb.log logs/enb.log
python parse_callflow.py
python serve_dashboard.py --port 9000 --no-open
```

## System topology

```
srsUE 4G  <-->  ZMQ IQ  <-->  srsENB  <-->  S1AP  <-->  srsEPC
```

Simulated LTE <span class="glossary-term" data-glossary-id="ue" data-glossary-term="UE" data-glossary-definition="User Equipment — the mobile device (phone/modem) that attaches to the cellular network." tabindex="0" role="button">UE</span>, <span class="glossary-term" data-glossary-id="enb" data-glossary-term="eNB" data-glossary-definition="Evolved Node B — the 4G LTE base station connecting UEs to the EPC over S1AP." tabindex="0" role="button">eNodeB</span>, and core. Signaling is produced by the real srsRAN
protocol stack; <span class="glossary-term" data-glossary-id="per-encoding" data-glossary-term="PER encoding" data-glossary-definition="Packed Encoding Rules — compact binary encoding of ASN.1 structures for RRC/NAS Layer-3 messages. Decoupling from PHY does not mean skipping PER encoding." tabindex="0" role="button">PER</span> reports can be extracted or reconstructed offline.

### Signaling message generation

Messages are generated by the real srsRAN stack — not approximated names or
templates. Offline tools can reconstruct or re-encode PER from captured traces
(see Missing fields below).

### Sending and receiving messages

**Radio side:**

```
srsUE RRC/NAS → PDCP/RLC/MAC/PHY → ZMQ IQ → srsENB PHY/MAC/RLC/PDCP/RRC
```

**Core side:**

```
srsENB → S1AP (SCTP) → srsEPC
```

### PER bytes to <span class="glossary-term" data-glossary-id="zmq-iq" data-glossary-term="ZMQ IQ" data-glossary-definition="Simulated radio link over network sockets where radio signals are represented as IQ samples with in-phase (I) and quadrature (Q) components." tabindex="0" role="button">ZMQ IQ samples</span>

1. **<span class="glossary-term" data-glossary-id="asn1" data-glossary-term="ASN.1" data-glossary-definition="A schema language for telecom messages: a strict protocol blueprint defining what messages exist, their fields, mandatory vs optional fields, choices, and value types." tabindex="0" role="button">ASN.1</span> → PER** — srsRAN holds C++ ASN.1 structures and packs them to PER
   (`asn1::bit_ref`, `msg.pack()`), producing a byte blob (logged as hex).
2. **PDCP** — wraps <span class="glossary-term" data-glossary-id="rrc" data-glossary-term="RRC" data-glossary-definition="Radio Resource Control — Layer 3 protocol between UE and base station for connection setup, mobility, and bearer configuration." tabindex="0" role="button">RRC</span> bytes for the radio bearer. SRB0 during early attach is
   often a transparent pipe; SRB1 adds sequence numbers, integrity, and
   encryption after security mode.
3. **RLC** — logical channel (SRB0 transparent mode for CCCH during <span class="glossary-term" data-glossary-id="rach" data-glossary-term="RACH" data-glossary-definition="Random Access Channel — procedure for a UE to request initial access to a cell (preamble, response, connection setup)." tabindex="0" role="button">RACH</span>; SRB1
   AM mode with segmentation after attach). Each SDU tagged with LCID.
4. **MAC** — schedules transport blocks (RACH Msg3/Msg4, <span class="glossary-term" data-glossary-id="harq" data-glossary-term="HARQ" data-glossary-definition="Hybrid Automatic Repeat Request — L1/L2 retransmission mechanism whose timing RRC procedures depend on." tabindex="0" role="button">HARQ</span>, timing advance).
5. **<span class="glossary-term" data-glossary-id="phy" data-glossary-term="PHY" data-glossary-definition="Physical layer — OFDM, modulation, and channel coding into IQ samples. A real DU must eventually demodulate IQ samples; PHY cannot be skipped entirely." tabindex="0" role="button">PHY</span>** — maps bits to symbols on <span class="glossary-term" data-glossary-id="prb" data-glossary-term="PRB" data-glossary-definition="Physical Resource Block — a unit of frequency-time resources on the LTE/NR grid allocated by the scheduler." tabindex="0" role="button">PRBs</span>, adds reference signals, <span class="glossary-term" data-glossary-id="ofdm" data-glossary-term="OFDM" data-glossary-definition="Orthogonal Frequency Division Multiplexing — splits bandwidth into parallel narrow sub-carriers, each carrying a slow stream. Resource mapping places IQ points on the OFDM time-frequency grid." tabindex="0" role="button">OFDM</span>
   modulates to time-domain **<span class="glossary-term" data-glossary-id="iq-samples" data-glossary-term="IQ samples" data-glossary-definition="Complex-valued in-phase and quadrature samples representing a modulated radio waveform digitally." tabindex="0" role="button">IQ samples</span>**. At this point the payload is IQ,
   not PER bytes.

## Deviations from a real system

### No real RF

No antenna, <span class="glossary-term" data-glossary-id="sdr" data-glossary-term="SDR" data-glossary-definition="Software-defined radio — programmable radio hardware controlled by software, used to transmit/receive real RF signals instead of ZMQ simulation." tabindex="0" role="button">SDR</span>, or <span class="glossary-term" data-glossary-id="fronthaul" data-glossary-term="Fronthaul Interface" data-glossary-definition="The connection between the RU and DU carrying low-level radio data, often IQ samples over protocols like eCPRI or Open Fronthaul." tabindex="0" role="button">RU fronthaul</span> — the radio link is **ZMQ IQ** between
`srsue` and `srsenb`.

### Simulated core

`srsEPC` replaces a real operator <span class="glossary-term" data-glossary-id="epc" data-glossary-term="EPC" data-glossary-definition="Evolved Packet Core — the 4G core network (MME, SGW, PGW, HSS) handling mobility, authentication, and sessions." tabindex="0" role="button">EPC</span>/5GC.

### Subscriber authentication

Test subscriber (`IMSI 302221000000001` with test <span class="glossary-term" data-glossary-id="ki" data-glossary-term="Ki" data-glossary-definition="Secret authentication key stored in the SIM and operator subscriber database." tabindex="0" role="button">Ki</span>/<span class="glossary-term" data-glossary-id="opc" data-glossary-term="OPc" data-glossary-definition="Operator-specific authentication value used with Milenage to prove the UE/SIM and network share the correct secret." tabindex="0" role="button">OPc</span>) instead of real
SIM/eSIM and <span class="glossary-term" data-glossary-id="hss-udm-provisioning" data-glossary-term="HSS/UDM provisioning" data-glossary-definition="Adding subscriber records (IMSI, keys, profiles) into the network subscriber database." tabindex="0" role="button">HSS/UDM provisioning</span>.

**Integration challenges for production parity:**

1. Real subscriber credentials or test SIM
2. Real frequency, bandwidth, timing, gain, and synchronization for RF/<span class="glossary-term" data-glossary-id="ru" data-glossary-term="RU" data-glossary-definition="Radio Unit — the physical antenna that talks directly with UEs. It converts analog RF to digital IQ samples and forwards them to the DU over ethernet, and modulates/demodulates U-Plane data per DU instructions." tabindex="0" role="button">RU</span>/SDR

## Missing fields and reconstruction

Captured traces in `22_decoded/` often lack fields required by the full LTE
ASN.1 structure. Example: `rrcConnectionSetupComplete` may only contain
`rrc-TransactionIdentifier` and `startList`, while <span class="glossary-term" data-glossary-id="pycrate" data-glossary-term="Pycrate" data-glossary-definition="Python ASN.1 toolkit used to load UL-DCCH-Message definitions and validate that reconstructed messages match the expected ASN.1 structure and fields." tabindex="0" role="button">Pycrate</span> expects the full
`UL-DCCH-Message` wrapper with `criticalExtensions`,
`rrcConnectionSetupComplete-r8`, `selectedPLMN-Identity`, and
`dedicatedInfoNAS`.

**Reconstruction** keeps existing trace fields and fills missing ones with
safe defaults:

| Field | Default | Meaning |
|---|---|---|
| `selectedPLMN-Identity` | `1` | First <span class="glossary-term" data-glossary-id="plmn" data-glossary-term="PLMN" data-glossary-definition="Public Land Mobile Network — identifies a mobile operator network via MCC (mobile country code) and MNC (mobile network code). Example: 302/221 is TELUS in Canada." tabindex="0" role="button">PLMN</span> from broadcast SIB1 |
| `dedicatedInfoNAS` | `00` | Placeholder when no full <span class="glossary-term" data-glossary-id="nas" data-glossary-term="NAS" data-glossary-definition="Non-Access Stratum — Layer 3 protocol between UE and core for attach, authentication, and session management." tabindex="0" role="button">NAS</span> attach payload |

### PER encoding (summary)

PER (Packed Encoding Rules) converts ASN.1 to compact binary: choice indices,
optional-field bitmaps, constrained integers, enums, and length-prefixed
strings — output as hex for inspection.

### Status classification

Dashboard metadata labels each message as exact, reconstructed, or partial
relative to the source trace.

## ML signaling generation

The ML generator is **not** wired into the live srsRAN ZMQ path or dashboard
replay. It produces standalone synthetic <span class="glossary-term" data-glossary-id="s1ap" data-glossary-term="S1AP" data-glossary-definition="S1 Application Protocol — a control-plane protocol between 4G eNB and EPC/MME handling communications between the base station and core network." tabindex="0" role="button">S1AP</span> logs for load and session-timing
experiments.

## UE signaling storm simulation

For scale beyond the 1–3 pair architecture, see the 5G twin's storm framework
in [`sa_signaling_storm.md`](sa_signaling_storm.md). The 4G twin's multi-pair
approach (`docker-compose.3ue.yml`, `demo3ue/`) measures host-level contention
between independent ZMQ links rather than simulating hundreds of UEs on one <span class="glossary-term" data-glossary-id="cell" data-glossary-term="Cell" data-glossary-definition="One carrier on one sector of a base station — a single radio coverage unit defined by frequency and physical cell ID. Multi-UE on one cell means many UEs contending on that single virtual cell." tabindex="0" role="button">cell</span>.

## Call flow (4G attach overview)

1. **Cell acquisition** — UE camps on eNB broadcast
2. **RACH** — PRACH preamble (Msg1), Random Access Response (Msg2)
3. **RRC connection** — Setup, Setup Complete (`UL-DCCH-Message`)
4. **NAS** — Attach Request inside `dedicatedInfoNAS`, authentication, bearer
   setup, Attach Complete
5. **Release** — eNB inactivity timer or explicit detach

The live dashboard (`serve_dashboard.py`) parses container logs into this
ladder with phase-ordered events — see [`lte_digital_twin.md`](lte_digital_twin.md).

## Protocol vs RF twin framing

This twin is a **protocol/signaling** twin, not an RF/channel twin. Fidelity
is in ASN.1 PER correctness and real stack behavior over ZMQ IQ, not in
propagation models or spectrum emulation. That separation is intentional: it
lets you validate signaling against real traces on commodity servers without
SDR hardware.
