# 4G srsTwin — Structure and Implementation

This chapter expands the 4G LTE twin documentation with system structure,
message flow, startup procedures, and known deviations from a production
network. It complements [`lte_digital_twin.md`](lte_digital_twin.md) and the
project [Glossary](../glossary.md).

## Startup

From `srsTwin/integration`:

```bash
docker compose -f docker-compose.4g.yml up -d --force-recreate
```

Wait ~30s, then verify attach:

```powershell
docker logs srstwin_ue4g 2>&1 | Select-String "Network attach successful"
docker logs srstwin_epc 2>&1 | Select-String "Attach request|UE Authentication Accepted|Attach Complete"
```

Dashboard (optional):

```bash
cd integration/dashboard
docker cp srstwin_ue4g:/tmp/ue4g.log logs/ue4g.log
docker cp srstwin_enb:/tmp/enb.log logs/enb.log
python parse_callflow.py
python serve_dashboard.py --port 9000 --no-open
```

## System topology

```
srsUE 4G  <-->  ZMQ IQ  <-->  srsENB  <-->  S1AP  <-->  srsEPC
```

Simulated LTE UE, eNodeB, and core. Signaling is produced by the real srsRAN
protocol stack; PER reports can be extracted or reconstructed offline.

### Signaling message generation

Messages are generated by the real srsRAN stack — not approximated names or
templates. Offline tools can reconstruct or re-encode PER from captured traces
(see Missing fields below).

### Sending and receiving messages

**Radio side:**

```
srsUE RRC/NAS → PDCP/RLC/MAC/PHY → ZMQ IQ → srsENB PHY/MAC/RLC/PDCP/RRC
```

**Core side:**

```
srsENB → S1AP (SCTP) → srsEPC
```

### PER bytes to ZMQ IQ samples

1. **ASN.1 → PER** — srsRAN holds C++ ASN.1 structures and packs them to PER
   (`asn1::bit_ref`, `msg.pack()`), producing a byte blob (logged as hex).
2. **PDCP** — wraps RRC bytes for the radio bearer. SRB0 during early attach is
   often a transparent pipe; SRB1 adds sequence numbers, integrity, and
   encryption after security mode.
3. **RLC** — logical channel (SRB0 transparent mode for CCCH during RACH; SRB1
   AM mode with segmentation after attach). Each SDU tagged with LCID.
4. **MAC** — schedules transport blocks (RACH Msg3/Msg4, HARQ, timing advance).
5. **PHY** — maps bits to symbols on PRBs, adds reference signals, OFDM
   modulates to time-domain **IQ samples**. At this point the payload is IQ,
   not PER bytes.

## Deviations from a real system

### No real RF

No antenna, SDR, or RU fronthaul — the radio link is **ZMQ IQ** between
`srsue` and `srsenb`.

### Simulated core

`srsEPC` replaces a real operator EPC/5GC.

### Subscriber authentication

Test subscriber (`IMSI 302221000000001` with test Ki/OPc) instead of real
SIM/eSIM and HSS/UDM provisioning.

**Integration challenges for production parity:**

1. Real subscriber credentials or test SIM
2. Real frequency, bandwidth, timing, gain, and synchronization for RF/RU/SDR

## Missing fields and reconstruction

Captured traces in `22_decoded/` often lack fields required by the full LTE
ASN.1 structure. Example: `rrcConnectionSetupComplete` may only contain
`rrc-TransactionIdentifier` and `startList`, while Pycrate expects the full
`UL-DCCH-Message` wrapper with `criticalExtensions`,
`rrcConnectionSetupComplete-r8`, `selectedPLMN-Identity`, and
`dedicatedInfoNAS`.

**Reconstruction** keeps existing trace fields and fills missing ones with
safe defaults:

| Field | Default | Meaning |
|---|---|---|
| `selectedPLMN-Identity` | `1` | First PLMN from broadcast SIB1 |
| `dedicatedInfoNAS` | `00` | Placeholder when no full NAS attach payload |

### PER encoding (summary)

PER (Packed Encoding Rules) converts ASN.1 to compact binary: choice indices,
optional-field bitmaps, constrained integers, enums, and length-prefixed
strings — output as hex for inspection.

### Status classification

Dashboard metadata labels each message as exact, reconstructed, or partial
relative to the source trace.

## ML signaling generation

The ML generator is **not** wired into the live srsRAN ZMQ path or dashboard
replay. It produces standalone synthetic S1AP logs for load and session-timing
experiments.

## UE signaling storm simulation

For scale beyond the 1–3 pair architecture, see the 5G twin's storm framework
in [`sa_signaling_storm.md`](sa_signaling_storm.md). The 4G twin's multi-pair
approach (`docker-compose.3ue.yml`, `demo3ue/`) measures host-level contention
between independent ZMQ links rather than simulating hundreds of UEs on one cell.

## Call flow (4G attach overview)

1. **Cell acquisition** — UE camps on eNB broadcast
2. **RACH** — PRACH preamble (Msg1), Random Access Response (Msg2)
3. **RRC connection** — Setup, Setup Complete (`UL-DCCH-Message`)
4. **NAS** — Attach Request inside `dedicatedInfoNAS`, authentication, bearer
   setup, Attach Complete
5. **Release** — eNB inactivity timer or explicit detach

The live dashboard (`serve_dashboard.py`) parses container logs into this
ladder with phase-ordered events — see [`lte_digital_twin.md`](lte_digital_twin.md).

## Protocol vs RF twin framing

This twin is a **protocol/signaling** twin, not an RF/channel twin. Fidelity
is in ASN.1 PER correctness and real stack behavior over ZMQ IQ, not in
propagation models or spectrum emulation. That separation is intentional: it
lets you validate signaling against real traces on commodity servers without
SDR hardware.
