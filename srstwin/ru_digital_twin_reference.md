## What the system is

Despite the name "<span class="glossary-term" data-glossary-id="ru" data-glossary-term="RU" data-glossary-definition="Radio Unit — the physical antenna that talks directly with UEs. It converts analog RF to digital IQ samples and forwards them to the DU over ethernet, and modulates/demodulates U-Plane data per DU instructions." tabindex="0" role="button">RU</span> Digital Twin," the target is really **two components**:

```
Component 1 — UE/RAN behavior generator
  Realistic UE signaling, RF conditions, mobility, handovers at scale

Component 2 — RU interface emulator
  Speaks eCPRI/Open Fronthaul toward O-DU so the DU/CU believes a real RU is attached
```

timing. Full RRC at thousands of UEs implies millions of concurrent timers.

```
gNB Emulator
├── Group 1 → normal registration
```

## Vendor specifics

Vendor quirks live at the **<span class="glossary-term" data-glossary-id="ru" data-glossary-term="RU" data-glossary-definition="Radio Unit — the physical antenna that talks directly with UEs. proprietary <span class="glossary-term" data-glossary-id="c-plane" data-glossary-term="C-Plane" data-glossary-definition="The control plane handles planning and instructions but does not carry actual user data. It contains commands for how the user and their data should be handled, receives real-time scheduling instructions (beamforming weights, PRB allocations) from the DU, and tells the RU what UE data to modulate or demodulate." tabindex="0" role="button">C-plane</span> sections) and **<span class="glossary-term" data-glossary-id="cu" data-glossary-term="CU" data-glossary-definition="Centralized Unit — higher layers (PDCP/RRC) in a split base station, less time-critical than the DU. Connects to DU over F1." tabindex="0" role="button">CU</span>/<span class="glossary-term" data-glossary-id="du" data-glossary-term="DU" data-glossary-definition="Distributed Unit — runs lower real-time layers (RLC, MAC, high-PHY scheduling) in an O-RAN split. A software DU runs on general-purpose servers; non-real-time simulation DUs can be time-dilated, real-time DUs driving real fronthaul cannot." tabindex="0" role="button">DU</span>** (scheduler assumptions, F1
extensions) — the latter is what you are testing, not emulating.

**Data-driven approach:** extract timing offsets, retry behavior, and error
handling from real traces → per-vendor profile plugins on a spec-compliant
<span class="glossary-term" data-glossary-id="ecpri" data-glossary-term="eCPRI" data-glossary-definition="Enhanced Common Public Radio Interface — a protocol that packages IQ samples into standard ethernet frames for efficient transport on fiber between RU and DU." tabindex="0" role="button">eCPRI</span> base layer. YANG models and trace formats are proprietary; models must
be **inferred from observed data**.
